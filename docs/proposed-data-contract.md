# Proposed Data Contract

Status: **Approved by owner on June 13, 2026.**

This proposal intentionally supports only the agreed Demo behavior:

- CardSN identifies an employee card.
- Permission is assigned directly between an employee and a bus.
- Android checks a locally cached permission list when offline.
- GPS recording runs continuously after device boot.

## 1. Conventions

- Entity primary keys use UUID unless a table explicitly defines a composite
  key. GPS points use `(device_id, sequence_no)`.
- API timestamps use ISO 8601 UTC, for example
  `2026-06-12T03:20:15Z`.
- Database timestamps use PostgreSQL `timestamptz`.
- CardSN is treated as an opaque, case-sensitive string. The system must not
  parse it as a number.
- CardSN should be normalized by trimming surrounding whitespace before it is
  saved or compared. No other transformation is proposed until actual reader
  output is tested.
- Latitude and longitude are API fields. PostgreSQL stores the position as a
  PostGIS `geography(Point, 4326)`.
- `created_at` and `updated_at` are server-managed fields.

## 2. Proposed Database Tables

### `bus`

| Field | Type | Null | Rule |
|---|---|---:|---|
| `id` | `uuid` | No | Primary key |
| `code` | `varchar(50)` | No | Unique human-readable bus identifier |
| `name` | `varchar(100)` | No | Display name |
| `active` | `boolean` | No | Default `true` |
| `permission_version` | `bigint` | No | Starts at `0`; current permission snapshot version |
| `created_at` | `timestamptz` | No | Server managed |
| `updated_at` | `timestamptz` | No | Server managed |

`permission_version` is incremented in the same database transaction whenever
the effective permission snapshot changes. This includes adding/removing a bus
permission and changing an assigned employee's `employee_no`, `name`,
`card_sn`, or `active` state.

The effective snapshot contains only active employees with a current
`bus_employee_permission` row. An inactive bus cannot authenticate its device
for normal synchronization.

### `device`

One Android device is assigned to one bus for the Demo.

| Field | Type | Null | Rule |
|---|---|---:|---|
| `id` | `uuid` | No | Primary key |
| `device_code` | `varchar(100)` | No | Unique provisioned identifier |
| `hardware_serial` | `varchar(100)` | No | Unique 618K SoC hardware identifier |
| `bus_id` | `uuid` | No | Foreign key to `bus` |
| `api_key_hash` | `varchar(255)` | Yes | Legacy per-device credential; not used by the shared-APK flow |
| `active` | `boolean` | No | Default `true` |
| `last_seen_at` | `timestamptz` | Yes | Last accepted device request |
| `created_at` | `timestamptz` | No | Server managed |
| `updated_at` | `timestamptz` | No | Server managed |

Only one active device may be assigned to a bus. Enforce this with a partial
unique index on `bus_id WHERE active = true`, rather than an unconditional
unique constraint.

Reinstalling the app does not create a new device identity. When hardware is
replaced:

1. Mark the old device inactive.
2. Register a new row with a new `id`, `device_code`, and `hardware_serial`.
3. Assign the new active device to the same bus.

### `employee`

| Field | Type | Null | Rule |
|---|---|---:|---|
| `id` | `uuid` | No | Primary key |
| `employee_no` | `varchar(50)` | No | Unique company employee number |
| `name` | `varchar(100)` | No | Display name |
| `card_sn` | `varchar(100)` | No | Unique NFC CardSN |
| `active` | `boolean` | No | Default `true` |
| `created_at` | `timestamptz` | No | Server managed |
| `updated_at` | `timestamptz` | No | Server managed |

### `bus_employee_permission`

| Field | Type | Null | Rule |
|---|---|---:|---|
| `bus_id` | `uuid` | No | Foreign key to `bus` |
| `employee_id` | `uuid` | No | Foreign key to `employee` |
| `created_at` | `timestamptz` | No | Server managed |

Primary key: (`bus_id`, `employee_id`).

Deleting this row removes permission. There is no shift, route, date range, or
temporary permission in the Demo.

### Deletion and history rules

- `bus`, `device`, and `employee` are not hard-deleted in the Demo; set
  `active = false` instead.
- A permission row may be deleted because it is current-state data, not
  history.
- `gps_point` and `boarding_event` foreign keys use restrictive deletion
  behavior so historical records cannot be orphaned.
- Deactivating a bus, device, or employee does not delete historical GPS or
  boarding events.

### `gps_point`

| Field | Type | Null | Rule |
|---|---|---:|---|
| `device_id` | `uuid` | No | Foreign key to `device` |
| `sequence_no` | `bigint` | No | Positive monotonic number generated by Android |
| `bus_id` | `uuid` | No | Foreign key to `bus` |
| `recorded_at` | `timestamptz` | No | Time measured by Android |
| `position` | `geography(Point,4326)` | No | Longitude/latitude |
| `accuracy_meters` | `real` | Yes | Android location accuracy |
| `received_at` | `timestamptz` | No | Time accepted by server |

Primary key: (`device_id`, `sequence_no`).

Android persists a 64-bit sequence counter and allocates the next value in the
same Room transaction that inserts the GPS row. Repeated uploads with the same
device and sequence are idempotent.

The counter is stored separately from queued GPS rows, starts at 1, and is
never recalculated from the queue because acknowledged or expired rows may
already have been deleted. Database constraint: `sequence_no > 0`.

`bus_id` records which bus owned the observation when it was captured. The
server derives both `device_id` and `bus_id` from authenticated device
assignment rather than accepting them from the upload payload.

If the app is reinstalled or its local data is cleared, the device must be
registered with a new `device_id` before it resumes tracking.

Proposed index: (`bus_id`, `recorded_at`).

A GiST index on `position` is intentionally omitted because the Demo does not
perform spatial-area queries. PostGIS still stores the point. Add a spatial
index only when a real spatial query requires it.

### `boarding_event`

Stores both successful and denied NFC scans for Demo verification.

| Field | Type | Null | Rule |
|---|---|---:|---|
| `id` | `uuid` | No | Primary key; generated by Android |
| `bus_id` | `uuid` | No | Foreign key to `bus` |
| `device_id` | `uuid` | No | Foreign key to `device` |
| `employee_id` | `uuid` | Yes | Null when CardSN is unknown |
| `card_sn` | `varchar(100)` | No | CardSN presented to device |
| `result` | `varchar(40)` | No | One of the result codes below |
| `scanned_at` | `timestamptz` | No | Time measured by Android |
| `permission_version` | `bigint` | Yes | Local permission version used |
| `received_at` | `timestamptz` | No | Time accepted by server |

Proposed result codes:

- `ALLOWED`
- `DENIED_UNKNOWN_CARD`
- `DENIED_NO_PERMISSION`
- `AUTH_DATA_NOT_READY`

Proposed index: (`bus_id`, `scanned_at`).

## 3. Android Local Data

Room keeps three logical stores:

### Permission cache

- `permission_version`
- `bus_id`
- `employee_id`
- `card_sn`
- `employee_no`
- `employee_name`
- `synced_at`

The full permission list is replaced atomically after a successful download.
It has no expiry for the Demo, so an offline bus can continue operating.

### Pending GPS queue

The local primary key is `sequence_no` because one Room database belongs to one
registered device. It contains the GPS fields sent to the server plus:

- `sync_status`: `PENDING` or `UPLOADED`
- `retry_count`
- `last_attempt_at`

Approved synchronization and retention rules:

- Read at most 500 pending GPS rows per batch, ordered by `sequence_no`
  ascending.
- Keep each request below 1 MB. If 500 rows would exceed that limit, send fewer
  rows.
- Load only the current batch from Room; never load the complete queue into
  memory.
- Delete a local row only after its sequence appears in the server's
  `acceptedSequenceNos`.
- Move a permanently rejected row to a local diagnostic/dead-letter state so
  one invalid record cannot block all later records.
- Retry failed uploads with exponential backoff.
- Keep at most 30 days of GPS rows on Android. When the limit is exceeded,
  remove the oldest rows and persist a dropped-row count for diagnostics.

### Pending boarding-event queue

Contains the boarding-event fields plus the same synchronization metadata.

Approved synchronization rules:

- Read at most 200 pending events per batch, ordered by `scanned_at` from
  oldest to newest.
- Keep each request below 1 MB.
- Delete a local row only after its id appears in the server's `acceptedIds`.
- Move a permanently rejected event to a local diagnostic/dead-letter state.
- Retry failed uploads with exponential backoff.

Only one synchronization worker may upload queued data at a time. After a
successful batch, it continues with the next batch until the queue is empty or
an upload fails.

## 4. Proposed Device API

Base path: `/api/device/v1`

Device authentication proposal:

```http
Authorization: Bearer <device-api-key>
X-Device-Hardware-Serial: <soc-model>-<soc-serial-hex>
```

All devices install the same APK and use the same deployment API key. The
hardware serial is unique and maps the request to a Device row and Bus. It is
an identifier, not a secret.

### Download permission snapshot

`GET /permissions`

```json
{
  "version": 18,
  "generatedAt": "2026-06-12T03:20:15Z",
  "bus": {
    "id": "218930ff-2bf3-4f9f-867c-97537a8e4132",
    "code": "BUS-01",
    "name": "Factory Bus 01"
  },
  "employees": [
    {
      "id": "9d5308a7-3394-4f68-8cd9-60ce2d9da91f",
      "employeeNo": "E00123",
      "name": "Demo Employee",
      "cardSn": "04A1B2C3D4"
    }
  ]
}
```

The `version` increases whenever the bus's effective permission list changes.
If the bus or authenticated device is inactive, the server rejects the request
instead of returning a snapshot.

### Upload GPS batch

`POST /gps-points/batch`

Maximum batch size: 500 points and 1 MB per request.

```json
{
  "points": [
    {
      "sequenceNo": 125081,
      "recordedAt": "2026-06-12T03:20:10Z",
      "latitude": 10.8231,
      "longitude": 106.6297,
      "accuracyMeters": 8.5
    }
  ]
}
```

```json
{
  "acceptedSequenceNos": [
    125081
  ],
  "rejected": [
    {
      "sequenceNo": 125082,
      "code": "INVALID_LOCATION",
      "retryable": false
    }
  ]
}
```

The server derives `bus_id` and `device_id` from the registered hardware serial
rather than trusting identifiers supplied in the request body.

An existing `(device_id, sequence_no)` with the same immutable payload is an
accepted idempotent retry. If the sequence already exists with different
`recordedAt`, coordinates, or accuracy, return non-retryable
`SEQUENCE_CONFLICT`; never overwrite the stored point.

Validation errors are returned per row in `rejected`. Request-level failures
return a non-2xx status, and Android retains the entire batch.

### Upload boarding-event batch

`POST /boarding-events/batch`

Maximum batch size: 200 events and 1 MB per request.

```json
{
  "events": [
    {
      "id": "b3ac02d7-2bb2-4efe-b1cb-dd8d37a8370a",
      "cardSn": "04A1B2C3D4",
      "employeeId": "9d5308a7-3394-4f68-8cd9-60ce2d9da91f",
      "result": "ALLOWED",
      "scannedAt": "2026-06-12T03:21:00Z",
      "permissionVersion": 18
    }
  ]
}
```

Example response:

```json
{
  "acceptedIds": [
    "b3ac02d7-2bb2-4efe-b1cb-dd8d37a8370a"
  ],
  "rejected": []
}
```

Boarding events retain UUID identity. Duplicate UUIDs with identical immutable
payload are accepted retries; conflicting payloads return non-retryable
`EVENT_ID_CONFLICT`. Validation errors use the same per-row `rejected`
structure with `id`, `code`, and `retryable`.

The event's `result`, `employeeId`, and `permissionVersion` describe the local
decision made at scan time. The backend stores this historical decision rather
than recalculating it using the current permission list. If `employeeId` is
present, it must reference an existing employee; it may no longer match the
employee's current CardSN because the offline snapshot can be older.

## 5. Proposed Admin API

Base path: `/api/admin/v1`

Initial operations:

- `GET /buses`
- `POST /buses`
- `PUT /buses/{busId}`
- `GET /employees`
- `POST /employees`
- `PUT /employees/{employeeId}`
- `GET /buses/{busId}/permissions`
- `PUT /buses/{busId}/permissions/{employeeId}`
- `DELETE /buses/{busId}/permissions/{employeeId}`
- `GET /buses/{busId}/gps-points?from=...&to=...`
- `GET /buses/{busId}/boarding-events?from=...&to=...`

For the Demo, route-history requests require `from` and `to`, allow at most a
24-hour interval, and return at most 10,000 points ordered by `recordedAt`.
This covers 8,640 expected points per day at a 10-second interval. Larger
ranges must be split into multiple requests.

Route history response:

```json
{
  "busId": "218930ff-2bf3-4f9f-867c-97537a8e4132",
  "from": "2026-06-12T00:00:00Z",
  "to": "2026-06-12T04:00:00Z",
  "points": [
    {
      "recordedAt": "2026-06-12T03:20:10Z",
      "latitude": 10.8231,
      "longitude": 106.6297,
      "accuracyMeters": 8.5
    }
  ]
}
```

## 6. Mapping Summary

| Concept | PostgreSQL | Device API | Android Room | Admin API |
|---|---|---|---|---|
| Card identity | `employee.card_sn` | `cardSn` | `card_sn` | `cardSn` |
| Bus permission | Join table | Permission snapshot | Permission cache | Permission endpoints |
| GPS identity | (`device_id`, `sequence_no`) | `sequenceNo` | `sequence_no` | Not exposed |
| GPS position | PostGIS `position` | Latitude/longitude | Latitude/longitude | Latitude/longitude |
| Scan outcome | `boarding_event.result` | `result` | Event queue | Boarding history |

## 7. Approved Demo Assumptions

The owner approved these assumptions on June 13, 2026:

1. Bus fields: `code`, `name`, and `active` are sufficient.
2. Employee fields: `employeeNo`, `name`, `cardSn`, and `active` are
   sufficient.
3. One Android device is permanently assigned to one bus for the Demo.
4. CardSN is stored as plaintext for the Demo but excluded from application
   logs.
5. The offline permission list does not expire during the Demo.
6. Both allowed and denied boarding scans are stored.
7. GPS keeps latitude, longitude, timestamp, and optional accuracy only.
8. Raw GPS and boarding-event data have no automatic deletion during the Demo.
9. Permission changes replace the device's complete local permission list.

## 8. Approved Queue Decisions

The owner approved these Android synchronization decisions on June 13, 2026:

- GPS upload: maximum 500 rows per batch.
- Boarding-event upload: maximum 200 rows per batch.
- Request payload: maximum 1 MB.
- Upload order: oldest pending rows first.
- Acknowledgement: delete GPS rows only by confirmed sequence numbers and
  boarding events only by confirmed UUIDs.
- Permanently rejected rows move to local dead-letter/diagnostic state and do
  not block later batches.
- Retry: exponential backoff after failure.
- Concurrency: one synchronization worker at a time.
- Memory: Room reads only one batch at a time.
- GPS idempotency: `(device_id, sequence_no)` makes retries safe.
- Boarding-event idempotency: Android-generated UUIDs make retries safe.
- Local GPS retention: maximum 30 days, dropping the oldest rows when needed.

The owner approved the GPS identity change on June 13, 2026:

- GPS has no standalone UUID.
- PostgreSQL primary key is (`device_id`, `sequence_no`).
- `sequence_no` is a 64-bit monotonically increasing value per device.
- GPS upload acknowledgement returns `acceptedSequenceNos`.
- App reinstall or local-data reset requires a new registered `device_id`.

The 30-day limit applies to the Android GPS queue only. It does not change the
server retention assumption in item 8 above.

## 9. Batch Validation Rules

- Latitude must be between `-90` and `90`.
- Longitude must be between `-180` and `180`.
- `accuracyMeters`, when present, must be finite and greater than or equal to
  zero.
- `recordedAt` and `scannedAt` must be valid UTC timestamps.
- A batch cannot contain the same GPS sequence or boarding-event UUID more
  than once.
- Permanent row errors are acknowledged through `rejected`; temporary server
  failures use a non-2xx response so the batch can be retried.
- Rows with `retryable = false` move to local dead-letter state. Rows with
  `retryable = true` remain pending.
